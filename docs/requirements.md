## 1. プロジェクト概要

### 1.1 目的

Test Insight Hubは、AI駆動開発における**独立したテスト監査ツール**として機能するElectronベースのローカルデスクトップツールである。プロジェクトのテストを自動実行してカバレッジを取得し、システム構造と対応付けて可視化し、テスト品質を分析して改善フィードバックを生成する。

本ツールの核心的な価値は「**書く側（プロジェクトAI）と検証する側（本ツール）の分離**」にある。AI駆動開発では同じモデルがコードとテストの両方を生成するため、盲点が生まれやすい。本ツールが独立した第三者の視点でテスト品質を評価し、具体的な改善指示をフィードバックすることで、この問題を解決する。

### 1.2 背景と課題

- AI駆動開発では同じモデルがコードとテストを書くため、**同一の盲点**を持ちやすい
- AIが生成するテストは「カバレッジ数値を上げるだけ」の浅いテスト（assert不足）になりがち
- テスト対象システムの全体構造が把握しにくく、テスト漏れが発生しやすい
- カバレッジレポートは数値のみで、システム構造との対応関係が直感的でない
- テスト不足箇所の特定から補完テスト作成までのワークフローが分断されている
- カバレッジ取得のための事前準備（テスト実行・レポート生成）がユーザー負担になっている

### 1.3 AI駆動開発におけるワークフロー比較

本ツールの立ち位置を明確にするため、3つのワークフローパターンを比較検討した。

#### パターンA: プロジェクト内完結型

```
プロジェクト側AI（Claude Code等）が全部やる:
  コード生成 → テスト生成 → テスト実行 → カバレッジ確認 → 修正
```

| 項目 | 評価 |
|------|------|
| スピード | 最速。AIが全部一気にやる |
| コンテキスト | AIがソースを全部見てるので精度が高い |
| 弱点 | **同じAIが書いて同じAIがレビュー**。盲点が生まれる |
| 弱点 | セッションが切れるとカバレッジの全体像が消える |
| 弱点 | 「カバレッジ上げて」→ AIがassert薄いテストを量産する傾向 |

#### パターンB: 外部ツール完結型

```
Test Insight Hub が全部やる:
  プロジェクト読み込み → テスト生成 → テスト実行 → 可視化 → フィードバック
```

| 項目 | 評価 |
|------|------|
| 独立性 | ソースとテストの評価が分離されている |
| 弱点 | **プロジェクト文脈が薄い**。ビジネスロジックの意図を知らない |
| 弱点 | 生成するテストが表面的になりやすい（形だけのテスト） |
| 弱点 | プロジェクト側のAIと二重作業になる |

#### パターンC: ハイブリッド型（本ツールの採用方式）

```
役割分担:
  プロジェクト側AI: コード + テストを書く（文脈を知っている）
  本ツール:         独立した目で検証 + 不足を指摘（監査役）
```

| 項目 | 評価 |
|------|------|
| 盲点の排除 | **書く側と検証する側が分離**。AI同士のクロスチェック |
| コンテキスト | テストはプロジェクト側AIが書く（文脈を知っている） |
| 可視化 | 全体像がセッション跨ぎで永続する |
| フィードバック | 具体的（どのファイル、どの関数、アサーション品質、優先度付き） |
| 二重作業 | なし。それぞれの役割が明確 |

#### 採用理由

パターンCを採用する。AI駆動開発の最大の問題は「同じモデルが書いて同じモデルがレビューする」ことであり、本ツールが独立した第三者として機能することで、この構造的問題を解決する。

### 1.4 ソリューション概要

```
┌────────────────────────────────────┐      ┌─────────────────────────────┐
│ プロジェクト側                      │      │ Test Insight Hub            │
│ (Claude Code / Cursor 等)          │      │ (独立監査ツール)             │
│                                    │      │                              │
│ 1. コードを書く                     │      │                              │
│ 2. テストを書く                     │      │                              │
│                    ─── 読み込み ──→ │      │ 3. プロジェクト解析           │
│                                    │      │    ・ソース構造 → ブロック図  │
│                                    │      │ 4. テスト自動実行             │
│                                    │      │    ・カバレッジ取得           │
│                                    │      │ 5. カバレッジ可視化           │
│                                    │      │    ・ブロック図に色分け       │
│                                    │      │ 6. テスト品質分析             │
│                                    │      │    ・アサーション深さ評価     │
│                                    │      │ 7. フィードバック生成          │
│ 9. FBを読んでテスト改善 ←── FB ──  │      │    ・不足箇所 + 品質指摘      │
│ 10. テスト再実行        ─── 再読込 → │     │ 8. 改善率トラッキング          │
└────────────────────────────────────┘      └─────────────────────────────┘
```

**本ツールの役割:**
- プロジェクト構造の可視化（ブロック図）
- テスト自動実行とカバレッジ取得（ユーザーの事前準備不要）
- カバレッジの可視化（ブロック図へのオーバーレイ）
- テスト品質の分析（アサーションの深さ評価）
- 不足箇所の特定と優先度付きフィードバック生成
- 改善サイクルのトラッキング

**本ツールがやらないこと:**
- テストコードの生成（プロジェクト側AIの仕事）
- ソースコードの修正（プロジェクト側AIの仕事）

### 1.5 プロジェクト構成モデル

本ツールは以下の2つのプロジェクト構成をサポートする。

**主ケース（同一リポジトリ構成）:**
ソースコードとテストコードが同一リポジトリ内に存在する構成。ユーザーはプロジェクトルートを1つ指定するだけで、ソースとテストの両方が読み込まれる。

```
project-root/
├── src/           ← ソースコード（F-001で解析）
│   └── ...
├── tests/         ← テストコード（F-003で解析）
│   └── ...
└── coverage/      ← カバレッジレポート（F-003で読み込み）
    └── lcov.info
```

- テストファイルの検出: `*.test.ts`, `*.spec.ts`, `test_*.py`, `*_test.go` 等の命名規約で自動判別
- テストディレクトリの検出: `tests/`, `test/`, `__tests__/`, `spec/` を自動認識
- ソースとテストのマッピング: カバレッジレポートに記載されたファイルパスで紐付ける

**副ケース（別リポジトリ構成）:**
ソースコードとテストコードが異なるディレクトリ（別リポジトリ等）に存在する構成。組込みシステム等で発生する。ユーザーはソースディレクトリとテストディレクトリをそれぞれ指定する。

- 初期リリースでサポートするが、同一リポジトリ構成を優先して設計・テストする
- 別リポジトリ構成では、ソースファイルパスとカバレッジレポート内パスの対応を手動設定で補正できる（パスプレフィックスのマッピング設定）

## 3. 機能要件

### 3.1 F-001: プロジェクト読み込み

**概要:** テスト対象プロジェクトのソースコードをローカルファイルシステムから読み込む

| 項目 | 内容 |
|------|------|
| 優先度 | 必須 (P0) |
| 入力 | プロジェクトのルートディレクトリパス |
| 出力 | パース済みのプロジェクト構造データ |

**詳細要件:**
- ユーザーがディレクトリ選択ダイアログからプロジェクトルートを指定できる
- プロジェクト構成の選択UI:
  - 初回読み込み時に「同一リポジトリ構成」（デフォルト）/ 「別リポジトリ構成」をラジオボタンで選択する
  - 同一リポジトリ構成: プロジェクトルートを1つ指定。ソースとテストは自動検出される
  - 別リポジトリ構成: ソースディレクトリとテストディレクトリをそれぞれ指定するフィールドを表示する
  - 選択した構成はプロジェクト設定（セクション9.3）に保存され、次回以降は自動適用される
- 以下のファイル種別を解析対象とする:
  - TypeScript (.ts, .tsx)
  - JavaScript (.js, .jsx)
  - Python (.py)
  - Go (.go)
  - Rust (.rs)
  - C/C++ (.c, .cpp, .h, .hpp)
- `.gitignore`、`node_modules`、`dist`、`build` 等の除外パターンを自動適用する
- 大規模プロジェクト（10,000ファイル以上）でも60秒以内に初回解析を完了する
- 解析結果をキャッシュし、変更ファイルのみ再解析する差分更新をサポートする

**受入基準:**
- [ ] ディレクトリ選択ダイアログからプロジェクトルートを指定し、解析が完了する
- [ ] 対応する6言語のファイルが正しく検出・解析される
- [ ] `.gitignore` に記載されたパターンのファイルが除外される
- [ ] 10,000ファイル（平均200行/ファイル）のプロジェクトで初回解析が60秒以内に完了する
- [ ] 2回目以降の読み込みでキャッシュが使用され、差分のみ解析される

**エラーハンドリング:**

| エラーケース | 振る舞い |
|------------|---------|
| 選択ディレクトリに読み取り権限がない | エラーダイアログ「アクセス権限がありません。ディレクトリの権限を確認してください」を表示 |
| 対応言語のソースファイルが0件 | 警告ダイアログ「対応するソースファイルが見つかりません。対応言語: TypeScript, JavaScript, Python, Go, Rust, C/C++」を表示 |
| シンボリックリンクによる循環参照を検出 | 循環参照パスをスキップし、ワーニングをログパネルに表示。解析は続行する |
| tree-sitterがパースエラーを返す（構文エラーのあるファイル） | 該当ファイルをスキップし、パースエラー一覧をログパネルに表示。他ファイルの解析は続行する |
| プロジェクトのファイル数が50,000を超える | 警告ダイアログ「大規模プロジェクトのため解析に時間がかかる可能性があります。続行しますか？」を表示。キャンセル可能 |

### 3.2 F-002: システムブロック図の可視化

**概要:** 読み込んだプロジェクトの構造をブロック図として描画する

| 項目 | 内容 |
|------|------|
| 優先度 | 必須 (P0) |
| 入力 | パース済みプロジェクト構造データ |
| 出力 | インタラクティブなブロック図 |

**詳細要件:**
- モジュール（ディレクトリ/パッケージ）単位でブロックを生成する
- モジュール間の依存関係を矢印で表現する
- 各ブロックに含まれるファイル数・関数数・行数を表示する
- ブロックをクリックすると内部構造を展開（ドリルダウン）できる
- ズーム・パン操作に対応する
- レイアウトアルゴリズム:
  - 階層レイアウト（デフォルト）
  - 力指向レイアウト（オプション）
- ブロック図をPNG/SVG形式でエクスポートできる

**受入基準:**
- [ ] モジュール単位のブロックが依存関係の矢印付きで描画される
- [ ] ブロックにファイル数・関数数・行数が表示される
- [ ] ブロッククリックで内部構造が展開（ドリルダウン）される
- [ ] ズーム・パン操作が動作する
- [ ] 階層レイアウトと力指向レイアウトを切り替えられる
- [ ] PNG/SVG形式でエクスポートしたファイルが正しく表示される
- [ ] 100モジュールのブロック図が2秒以内にレンダリングされる

**エラーハンドリング:**

| エラーケース | 振る舞い |

### 3.3 F-003: テスト自動実行とカバレッジ取得

**概要:** 対象プロジェクトのテストフレームワークを自動検出し、ツール内からテストを実行してカバレッジレポートを取得する。ユーザーの事前準備（手動でのテスト実行やレポート生成）を不要にする。

| 項目 | 内容 |
|------|------|
| 優先度 | 必須 (P0) |
| 入力 | プロジェクトルートパス（F-001で指定済み） |
| 出力 | カバレッジデータ、テスト実行結果 |

**詳細要件:**

- テストフレームワークの自動検出:
  - `package.json` に `vitest` or `jest` → `npx vitest run --coverage --reporter=json` or `npx jest --coverage --coverageReporters=json`
  - `pytest.ini` / `setup.cfg` / `pyproject.toml` に pytest → `pytest --cov --cov-report=json`
  - `go.mod` → `go test -coverprofile=cover.out ./...`
  - `Cargo.toml` → `cargo llvm-cov --json`
  - 検出できない場合、ユーザーにテストコマンドの手動入力を求める
- テスト実行はプロジェクトルートで `child_process.exec` を使用する
- テスト実行中の進捗（stdout/stderr）をログパネルにリアルタイム表示する
- テスト実行完了後、生成されたカバレッジレポートを自動的に読み込む
- カバレッジレポートの読み込みは以下の形式に対応:
  - Vitest / Jest (Istanbul JSON形式)
  - pytest (coverage.py JSON形式)
  - Go (go test -coverprofile形式)
  - Rust (cargo-llvm-cov JSON形式)
  - gcov/lcov (lcov.info形式)
- テストファイルを解析し、テストケース一覧とテスト種別（Unit/Integration/E2E）を分類する
- **既存レポートの直接読み込みも引き続きサポート**する（手動指定またはプロジェクト内の自動検出）
- テスト実行タイムアウト: デフォルト300秒（設定で変更可能）

**受入基準:**
- [ ] package.json から Vitest/Jest を自動検出し、テスト実行とカバレッジ取得が完了する
- [ ] pyproject.toml から pytest を自動検出し、テスト実行とカバレッジ取得が完了する
- [ ] go.mod から Go テストを自動検出し、テスト実行とカバレッジ取得が完了する
- [ ] Cargo.toml から Rust テストを自動検出し、テスト実行とカバレッジ取得が完了する
- [ ] テスト実行中の stdout/stderr がログパネルにリアルタイム表示される
- [ ] テスト実行完了後、カバレッジデータが自動で読み込まれる
- [ ] フレームワーク検出に失敗した場合、手動コマンド入力ダイアログが表示される
- [ ] 既存のカバレッジレポートファイルを手動で指定して読み込める
- [ ] テスト実行がタイムアウトした場合にエラー通知される

**エラーハンドリング:**

| エラーケース | 振る舞い |
|------------|---------|
| テストフレームワークを検出できない | 「テストフレームワークを自動検出できませんでした。テスト実行コマンドを入力してください」とコマンド入力ダイアログを表示 |
| テスト実行が失敗（exit code != 0） | テスト失敗の詳細をログパネルに表示。一部テスト失敗でもカバレッジレポートが生成されていれば読み込みを試行する |
| テスト実行がタイムアウト | 「テスト実行がタイムアウトしました（300秒）。設定でタイムアウト時間を延長するか、テスト対象を絞ってください」と表示 |
| カバレッジレポートが生成されない | 「テストは完了しましたが、カバレッジレポートが見つかりません。テストフレームワークのカバレッジ設定を確認してください」と表示 |
| カバレッジレポートのフォーマットが不正・破損 | エラーメッセージ「カバレッジレポートの読み込みに失敗しました。検出された形式: [形式名]」を表示 |
| カバレッジレポートとソースコードの不一致 | 不一致ファイルをワーニング表示。一致率50%未満で警告 |

### 3.4 F-004: テストカバレッジの可視化

**概要:** ブロック図上にカバレッジ情報をオーバーレイ表示する

| 項目 | 内容 |
|------|------|
| 優先度 | 必須 (P0) |
| 入力 | ブロック図データ + カバレッジデータ |
| 出力 | カバレッジ色付きブロック図、詳細パネル |

**詳細要件:**
- 各ブロック（モジュール）のカバレッジ率に応じて色分けする:
  - 80%以上: 緑
  - 50〜79%: 黄
  - 50%未満: 赤
  - テスト未実施: グレー
- カバレッジ表示モードを切り替えられる:
  - 行カバレッジ（Line Coverage）
  - 分岐カバレッジ（Branch Coverage）
  - 関数カバレッジ（Function Coverage）
- ブロッククリックで詳細パネルを表示:
  - ファイル別カバレッジ一覧
  - カバーされていない関数・行のリスト
  - 対応するテストケースの一覧
- テスト種別（Unit/Integration/E2E）ごとのフィルタリング
- カバレッジのサマリーダッシュボード:
  - プロジェクト全体のカバレッジ率
  - モジュール別カバレッジランキング
  - 前回スキャンとの差分（改善/悪化）

```
┌─────────────────────────────────────────────────────────────┐
│  カバレッジビュー                    [Line▼] [Unit|Int|E2E] │
│                                                              │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐               │
│   │  api/   │───→│ service/│───→│  repo/  │               │
│   │  92%    │    │  67%    │    │  45%    │               │
│   │ ██████░ │    │ ████░░░ │    │ ███░░░░ │               │
│   └─────────┘    └─────────┘    └─────────┘               │
│   (緑)           (黄)           (赤)                        │
│                                                              │
│  ─────────────── 詳細パネル ───────────────                  │
│  repo/ (45%)                                                 │
│  ├── user-repo.ts      38%  ← 未カバー: L23-40, L55-62     │
│  ├── order-repo.ts     52%  ← 未カバー: L10-18             │
│  ├── product-repo.ts   48%  ← 未カバー: L30-45             │
│  └── base-repo.ts      42%  ← 未カバー: L5-20              │
│                                                              │
│  未カバー関数: findByFilter, bulkInsert, softDelete          │
└─────────────────────────────────────────────────────────────┘
```

**受入基準:**
- [ ] 各ブロックがカバレッジ率に応じて色分け（緑/黄/赤/グレー）される
- [ ] Line/Branch/Functionの3種類のカバレッジモード切替が動作する
- [ ] ブロッククリックで詳細パネルが1秒以内に表示される
- [ ] 詳細パネルにファイル別カバレッジ、未カバー関数・行、対応テストケースが表示される
- [ ] テスト種別（Unit/Integration/E2E）でフィルタリングが動作する
- [ ] サマリーダッシュボードにプロジェクト全体のカバレッジ率・モジュール別ランキング・前回差分が表示される
- [ ] カバレッジ0%のモジュールがグレーで表示される

**エラーハンドリング:**

| エラーケース | 振る舞い |
|------------|---------|
| カバレッジデータが読み込まれていない状態でカバレッジビューを開く | 「カバレッジデータが読み込まれていません」とガイド付きメッセージを表示し、F-003の読み込みフローへ誘導 |
| 分岐カバレッジが含まれないレポート形式でBranchモードを選択 | 「このカバレッジレポートには分岐カバレッジデータが含まれていません。Line Coverageモードで表示します」と通知しフォールバック |

### 3.5 F-005: フィードバックファイルの生成

**概要:** カバレッジ不足箇所を分析し、フィードバックファイルを生成する

| 項目 | 内容 |
|------|------|
| 優先度 | 必須 (P0) |
| 入力 | カバレッジ分析結果 |
| 出力 | フィードバックファイル（JSON形式） |

**詳細要件:**
- カバレッジ閾値（デフォルト80%）を下回るモジュール・ファイルを検出する
- 閾値はプロジェクト設定で変更可能とする
- フィードバックファイルに以下の情報を含める:
  - 不足箇所のファイルパスと行範囲
  - 未テストの関数・メソッド一覧
  - 推奨テスト種別（Unit/Integration/E2E）
  - 優先度スコア（以下の計算式で算出）
  - 生成日時とツールバージョン

**優先度スコア計算仕様:**

```
priorityScore = (coverageGapWeight × coverageGap)
              + (complexityWeight  × complexity)
              + (changeFreqWeight  × changeFrequency)

デフォルト重み: coverageGapWeight=0.4, complexityWeight=0.3, changeFreqWeight=0.3
スコア範囲: 0〜100（正規化済み）
```

各因子の定義:

| 因子 | 算出方法 | フォールバック |
|------|---------|--------------|
| `coverageGap` | `(targetCoverage - currentCoverage) / targetCoverage × 100` で正規化 | — |
| `complexity` | 循環的複雑度（Cyclomatic Complexity）をtree-sitterのASTから算出。関数単位で計測し、ファイル単位は関数の最大値を採用 | AST解析不能の場合、行数ベースの簡易推定（LOC / 10、上限100） |
| `changeFrequency` | git logから直近90日間のコミット数を集計。対象ファイルに変更を含むコミットをカウント | gitリポジトリでない場合は `0`（変更頻度不明）とし、残り2因子で正規化。ファイルの最終更新日時が30日以内なら `50`、それ以外は `0` とする簡易モードも提供 |

- 重み係数はプロジェクト設定で変更可能とする（上級者向け、設定画面の「詳細設定」セクション）
- `priority` ラベルは `priorityScore` から自動判定: `high`（70以上）, `medium`（40〜69）, `low`（39以下）
- フィードバックファイルのフォーマット:

```json
{
  "version": "1.0.0",
  "generatedAt": "2026-03-22T10:30:00Z",
  "projectRoot": "/path/to/project",
  "coverageThreshold": 80,
  "summary": {
    "totalModules": 12,
    "belowThreshold": 4,
    "totalUncoveredFunctions": 23,
    "overallCoverage": 68.5
  },
  "gaps": [
    {
      "filePath": "src/repo/user-repo.ts",
      "moduleName": "repo",
      "currentCoverage": 38,
      "targetCoverage": 80,
      "uncoveredLines": [
        { "start": 23, "end": 40, "functionName": "findByFilter" },
        { "start": 55, "end": 62, "functionName": "softDelete" }
      ],
      "recommendedTestType": "unit",
      "priority": "high",
      "priorityScore": 87,
      "complexity": 12,
      "changeFrequency": 8
    }
  ],
  "recommendations": [
    {
      "type": "unit",
      "targetFile": "src/repo/user-repo.ts",
      "suggestedTestFile": "src/repo/__tests__/user-repo.test.ts",
      "functions": ["findByFilter", "softDelete"],
      "description": "findByFilterとsoftDeleteの境界値テストとエラーケーステストを追加"
    }
  ]
}
```

**受入基準:**
- [ ] カバレッジ閾値（デフォルト80%）以下のモジュール・ファイルが検出される
- [ ] 閾値をプロジェクト設定から変更でき、変更後の閾値で再検出が動作する
- [ ] フィードバックJSONが上記フォーマットに準拠して生成される
- [ ] 各gapに優先度スコア（priorityScore）が計算されている
- [ ] recommendationsに推奨テストファイルパスと関数名が含まれている
- [ ] フィードバック生成が5秒以内に完了する

**エラーハンドリング:**

| エラーケース | 振る舞い |
|------------|---------|
| カバレッジデータが未読み込み状態でFB生成を実行 | 「カバレッジデータを先に読み込んでください」とガイド付きメッセージを表示 |
| 全モジュールが閾値以上（不足箇所なし） | 「すべてのモジュールがカバレッジ閾値（80%）を満たしています。フィードバックファイルは生成されませんでした」と通知 |
| フィードバック生成中にソースファイルが変更された | 「ソースファイルが変更されました。最新の状態で再解析してからフィードバックを生成してください」と警告し、生成を中断 |

### 3.7 F-007: フィードバック適用と再検証

**概要:** フィードバックに基づきテストが追加された後、再度カバレッジを検証する

| 項目 | 内容 |

| 優先度 | 重要 (P1) |
| 入力 | 更新後のテストコード、前回フィードバック |
| 出力 | 改善状況レポート |

**詳細要件:**
- 前回のフィードバックファイルと最新カバレッジを比較する

- 改善率をサマリーとして表示する
- フィードバック→テスト追加→再検証のサイクルをトラッキングする

**受入基準:**
- [ ] 前回FBと最新カバレッジの比較で、改善箇所（緑化）と未改善箇所が区別して表示される
- [ ] 改善率がサマリーとして数値表示される
- [ ] 複数回のフィードバックサイクルの履歴がトラッキングされ、一覧表示される
- [ ] 前回FBで指摘された関数のうち、テストが追加された関数が特定される

**エラーハンドリング:**

| エラーケース | 振る舞い |

### 3.8 F-008: テスト品質分析

**概要:** テストコードを静的解析し、アサーションの深さ・網羅性をスコア化する。カバレッジ（行の実行率）だけでは検出できない「見かけ倒しのテスト」を特定する。AI駆動開発において、AIが生成した浅いテスト（ステータスコードだけチェック等）を検出するための機能。

| 項目 | 内容 |
|------|------|
| 優先度 | 重要 (P1) |
| 入力 | テストファイル群 |
| 出力 | テスト品質スコア、品質指摘の一覧 |

**詳細要件:**

- テストファイルを静的解析し、各テストケースのアサーション品質をスコア化する
- 品質スコアの算出基準:

| チェック項目 | 検出パターン例 | スコア |
|------------|-------------|--------|
| ステータスコード検証 | `expect(res.status).toBe(200)` | +1 |
| レスポンスボディ検証 | `expect(res.body)...` | +2 |
| プロパティ存在検証 | `toHaveProperty`, `toBeDefined` | +2 |
| 値の型・形式検証 | `toBeInstanceOf`, `toMatch`, `toBeGreaterThan` | +3 |
| エラーケーステスト | `expect(res.status).toBe(4xx/5xx)`, `toThrow` | +3 |
| 境界値テスト | 同一関数に対して複数パターンの入力値 | +3 |

- ファイル単位・関数単位で品質スコアを集計する
- 品質レベルを判定:
  - HIGH: スコア70%以上（十分なアサーション）
  - MEDIUM: スコア40〜69%（基本的だが不十分）
  - LOW: スコア40%未満（見かけ倒し）
- フィードバック（F-005）に品質スコアを統合し、「カバレッジは高いが品質が低い」ケースも指摘する

**品質分析の具体例:**

```
src/api/handler.test.ts:
  test "GET /users returns list"
    アサーション:
      ✅ ステータスコード検証 (+1)
      ❌ レスポンスボディ未検証
      ❌ プロパティ未検証
      ❌ エラーケースなし
    品質スコア: 1/14 (7%) → LOW

    指摘: 「レスポンスボディの構造検証とエラーケース（404等）のテストを追加してください」

  test "POST /users creates user"
    アサーション:
      ✅ ステータスコード検証 (+1)
      ✅ レスポンスボディ検証 (+2)
      ✅ プロパティ検証 (+2)
      ✅ エラーケース検証 (+3)
    品質スコア: 8/14 (57%) → MEDIUM
```

**受入基準:**
- [ ] テストファイルからアサーションパターンが検出される
- [ ] 各テストケースに品質スコアが算出される
- [ ] HIGH/MEDIUM/LOW の品質レベルが正しく判定される
- [ ] 品質スコアがフィードバック（F-005）のgapsとrecommendationsに統合される
- [ ] 「カバレッジ80%だが品質LOW」のケースがフィードバックで指摘される

**エラーハンドリング:**

| エラーケース | 振る舞い |
|------------|---------|
| テストファイルの解析に失敗 | 該当ファイルの品質スコアを「不明」としてスキップ。他ファイルの分析は続行 |
| アサーションパターンが検出されない言語 | 「この言語のアサーション品質分析は未対応です。カバレッジスコアのみで評価します」と通知 |

### 4.1 NF-001: プラットフォーム対応

| 項目 | 内容 |
|------|------|

| 指標 | 目標値 |
|------|--------|

| カバレッジオーバーレイ適用 | 1秒以内 |
| フィードバックファイル生成 | 5秒以内 |

### 4.3 NF-003: ユーザビリティ

- ツールチップ・ヘルプテキストによるガイダンスを提供する
- 初回起動時にオンボーディングウィザードを表示する
- キーボードショートカットに対応する
- ダークモード/ライトモードを切り替えられる
- 日本語/英語のUI言語切替をサポートする

- ローカルファイルシステムへのアクセスはユーザーが明示的に選択したディレクトリのみとする

- 設定ファイルにシークレット情報を保存しない
- Electronのコンテキスト分離（contextIsolation）を有効にする
- nodeIntegrationはレンダラープロセスで無効にする

### 4.5 NF-005: 拡張性

- プラグインアーキテクチャにより新しい言語パーサーを追加可能とする
- カバレッジレポート形式の追加をプラグインで対応可能とする
- フィードバックファイルフォーマットをバージョニングし、後方互換性を維持する

### 6.2 IPCチャネル一覧

| `project:parse-error` | Main→Renderer | 解析中のエラー・ワーニングを通知 | F-001 |

```
         │
         ↓

    ├── レポート形式の自動判定

    └── モジュール単位で集約
         │

         │
         ↓
[7] ユーザーがフィードバック生成を実行
         │
         ↓
[8] Feedback Generator がフィードバックを生成
    ├── 閾値以下のモジュール・関数を検出
    ├── 優先度スコアを計算
    └── フィードバックJSON生成
         │
         ↓
[9] Feedback View が内容を表示、ユーザーが配置を確認
         │
         ↓
[10] フィードバックファイルをプロジェクトに配置
         │
         ↓
[11] 対象プロジェクト側でFBを読み込み、テスト追加
         │
         ↓

```

## 9. データ永続化仕様

### 9.1 保存先

アプリケーションデータは Electron の `app.getPath('userData')` 配下に保存する。

| OS | パス例 |
|----|--------|
| Windows | `%APPDATA%/TestInsightHub/` |
| macOS | `~/Library/Application Support/TestInsightHub/` |
| Linux | `~/.config/TestInsightHub/` |

### 9.2 データ種別と保存仕様

| データ種別 | ファイルパス | フォーマット | ライフサイクル |
|-----------|-------------|-------------|--------------|
| グローバル設定 | `settings.json` | JSON | アプリ削除まで永続 |
| プロジェクト設定 | `projects/{projectId}/config.json` | JSON | ユーザーが明示的に削除するまで永続 |
| プロジェクト履歴 | `recent-projects.json` | JSON | 最大20件。LRU方式で古いものを自動削除 |
| 解析キャッシュ | `projects/{projectId}/cache/` | バイナリ（MessagePack） | プロジェクト再読み込み時に自動更新。手動クリア可能 |
| フィードバック履歴 | `projects/{projectId}/feedback-history/` | JSON（タイムスタンプ付き） | 最大50件。古いものを自動削除 |
| ウィンドウ状態 | `window-state.json` | JSON | アプリ終了時に自動保存 |

### 9.3 プロジェクト設定ファイルのスキーマ

```json
{
  "projectId": "sha256-of-absolute-path",
  "projectName": "my-project",
  "rootPath": "/path/to/project",
  "testRootPath": null,
  "coverageReportPath": null,
  "coverageThreshold": 80,
  "colorThresholds": {
    "green": 80,
    "yellow": 50
  },
  "excludePatterns": [],
  "pathMappings": [],
  "lastOpenedAt": "2026-03-22T10:30:00Z"
}
```

- `testRootPath`: 別リポジトリ構成の場合のみ設定。`null` の場合は同一リポジトリ構成として `rootPath` 内を検索
- `pathMappings`: 別リポジトリ構成でカバレッジレポート内パスとソースパスの対応を定義

### 9.4 キャッシュ管理

- キャッシュの最大サイズ: プロジェクトあたり500MB（超過時は古いキャッシュから削除）
- 全プロジェクト合計の最大サイズ: 2GB（設定画面で変更可能）
- 設定画面からキャッシュの手動クリアが可能
- キャッシュが破損した場合は自動検出し、再構築を実行

### 9.5 設定の優先度

グローバル設定とプロジェクト設定が競合する場合、**プロジェクト設定が優先**される。

```
グローバル設定（settings.json）
  ↓ 上書き
プロジェクト設定（projects/{id}/config.json）
```

## 10. 用語定義

| 用語 | 定義 |
|------|------|
| ブロック図 | モジュール（ディレクトリ/パッケージ）を矩形ブロックとして描画し、依存関係を矢印で結んだ図 |
| カバレッジ | テストによって実行されたコード（行・分岐・関数）の割合。テスト実行による動的計測で取得 |
| テスト品質スコア | テストコードのアサーション深さ・網羅性を静的解析でスコア化した指標。カバレッジとは独立 |
| フィードバックファイル (FB) | テストカバレッジの不足箇所・品質指摘・推奨アクションをJSON形式で記述したファイル |
| フィードバックサイクル | テスト実行→カバレッジ＋品質分析→FB生成→プロジェクト側AIがテスト改善→再検証のループ |
| モジュール | プロジェクト内のディレクトリまたはパッケージを1つの単位として扱ったもの |
| プロジェクト側AI | Claude Code、Cursor等、対象プロジェクト内でコードとテストを生成するAIツール |
| 独立監査 | コードを書いた主体とは異なる主体がテスト品質を検証すること。AI駆動開発の盲点を排除する |

## 11. 制約事項

1. **オフライン動作必須:** インターネット接続なしで全機能が動作すること
2. **データ外部送信禁止:** プロジェクトのソースコード・テストデータを外部サーバーに送信しないこと
3. **非破壊的操作:** 対象プロジェクトのソースコードを変更しないこと（フィードバックファイルの配置のみ許可）
4. **プラグインサンドボックス:** サードパーティプラグインはサンドボックス内で実行すること

---

## 12. 前提条件

1. ユーザーのマシンにNode.js 20以上がインストールされていること（Electronアプリ配布時は不要）
2. 対象プロジェクトのソースコードがローカルファイルシステム上に存在すること
3. 対象プロジェクトのテストフレームワーク（Vitest, Jest, pytest, go test, cargo test等）の実行環境がインストール済みであること
4. テストカバレッジレポートの事前生成は不要（本ツールがテスト実行とレポート生成を行う）
